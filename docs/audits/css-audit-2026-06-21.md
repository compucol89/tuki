## 4. Anomalías Detectadas

### 4.1 🔴 CRÍTICO: Bootstrap version mismatch

| Ubicación | Versión | Archivo |
|-----------|---------|---------|
| Frontend | **4.5.3** | `public/assets/front/css/bootstrap.4.5.3.min.css` |
| Admin/Organizer | **4.3.1** | `public/assets/admin/css/bootstrap.min.css` |

**Evidencia**: 
- Frontend: `head -1` muestra `Bootstrap v4.5.3`
- Admin: `head -1` muestra `Bootstrap v4.3.1`

**Impacto**: Comportamientos divergentes en grid, spacing utilities, modals, tooltips, y componentes que se usan en ambos lados. Riesgo de bugs visuales sutiles.

**Recomendación**: Unificar a Bootstrap 4.5.3 en admin. Reemplazar `bootstrap.min.css` en admin con la versión 4.5.3.

### 4.2 🔴 CRÍTICO: main.css tiene CERO selectores únicos vs atlantis.css

**Evidencia**: 
```bash
comm -23 <(selectores main.css) <(selectores atlantis.css) | wc -l
# Resultado: 0
```

**Análisis**: `main.css` (8,909 líneas, 1,374 selectores) no contiene ningún selector que no esté ya presente en `atlantis.css` (14,683 líneas, 2,605 selectores). Esto significa que main.css es técnicamente un subset/extensión de atlantis.css que carga propiedades adicionales para los mismos selectores. Ambos archivos se cargan en secuencia: atlantis.css primero, luego main.css lo pisa.

**Impacto**: 23,592 líneas de CSS cargadas donde el override se podría consolidar. Duplica el peso de CSS para los mismos componentes.

**Stop condition parcial**: Fusionar requiere testing visual exhaustivo — main.css probablemente contiene overrides críticos de TukiPass sobre Atlantis. No eliminar sin verificación.

**Recomendación**: Fase 2 (consolidación con QA visual). Extraer solo las reglas diferentes de main.css e integrarlas como overrides específicos.

### 4.3 🔴 CRÍTICO: website-color.php — Dead code con riesgo de seguridad

**Evidencia**:
```bash
grep -rl "website-color" resources/views/ app/
# Resultado: (vacío — 0 referencias)
```

**Archivo**: `public/assets/admin/css/website-color.php` (605 líneas)
**Contenido**: Script PHP que recibe colores vía `$_GET` y genera CSS dinámico. Sin sanitización de entrada. Sin autenticación.
**Selectores generados**: `.spinner`, `.page-item`, `.header-area`, `.banner-area`, `.dream-course-area`, `.footer-area`, etc. — todos de un tema LMS legacy (cursos).

**Impacto**: 
- Dead code (0 referencias en vistas/controllers)
- Si es accesible vía web, es un vector de ataque (CSS injection vía query params)
- Los selectores que genera no coinciden con el markup actual de TukiPass

**Recomendación**: **Eliminar inmediatamente** (Fase 1). Los colores dinámicos ya se manejan vía `--primary-color` en el `<style>` inline de `styles.blade.php` y `admin-skin.css`.

### 4.4 🟠 ALTO: !important abuse — 790 en atlantis.css

**Evidencia**:
```
790 !important: atlantis.css
188 !important: admin-skin.css
80  !important: style.css
73  !important: event-detail.css
73  !important: admin-main.css
63  !important: organizer.css
63  !important: events.css
```

**Análisis**: `atlantis.css` promedia 1 `!important` cada 18.5 líneas. Esto es un antipatrón de especificidad que hace que los overrides en `admin-skin.css` también requieran `!important` (188 ocurrencias), creando una guerra de especificidad.

**Impacto**: Cualquier cambio futuro en admin-skin.css requiere `!important` para ganar la cascada, perpetuando el problema.

**Recomendación**: No se puede resolver sin refactorizar atlantis.css (que es third-party). En fase 3, evaluar cargar admin-skin.css DESPUÉS de atlantis.css y main.css (ya lo hace) y usar selectores de mayor especificidad en lugar de `!important`.

### 4.5 🟠 ALTO: 4 archivos render-blocking en backend

**Evidencia**: `styles.blade.php` backend líneas 13, 44, 47, 50 cargan sin `media="print"`:
- `bootstrap.min.css` (línea 13)
- `atlantis.css` (línea 44) — 14,683 líneas
- `admin-main.css` (línea 47) — 772 líneas
- `admin-skin.css` (línea 50) — 612 líneas

**Impacto**: ~24,000 líneas de CSS bloquean el primer render del panel admin.

**Recomendación**: Agregar `media="print" onload="this.media='all'"` con `<noscript>` fallback. Bajo riesgo en admin ya que es un panel interno, pero mejora la percepción de velocidad.

### 4.6 🟠 ALTO: IE y Firefox legacy en atlantis.css

**Evidencia**:
```
Líneas 2210, 2242: /* IE 10+ */
Líneas 2200, 2232: /* Chrome/Opera/Safari */
Líneas 2205, 2237: /* Firefox 19+ */
Líneas 2215, 2247: /* Firefox 18- */
```
- 92 ocurrencias de prefijos legacy (`-ms-`, `-o-`, `-moz-`)

**Impacto**: Código muerto que añade peso innecesario. IE está fuera de soporte desde 2022. Firefox 18- es de 2013.

**Recomendación**: No modificar atlantis.css directamente (third-party). Si se hace un fork, eliminar secciones IE/Firefox legacy. Ahorro estimado: ~200-500 líneas.

### 4.7 🟡 MEDIO: all.min.css — Archivo huérfano

**Evidencia**: 
```bash
grep -rl "all.min.css" resources/views/ app/
# Resultado: (vacío)
```
- MD5 de `all.min.css`: `b2930cbbaa3187ac98ef85f31ec2bcb6`
- MD5 de `fontawesome.css`: `5f9fddfbd8338862c683180ecf9440d4`
- **No son el mismo archivo** — all.min.css es una versión anterior/diferente de Font Awesome

**Recomendación**: Eliminar de `public/assets/admin/css/all.min.css`. No está referenciado en ninguna vista.

### 4.8 🟡 MEDIO: Breakpoints inconsistentes

**Evidencia** — 40+ variaciones de media queries detectadas:

| Breakpoint | Usos | Archivos |
|------------|------|----------|
| `(max-width: 767px)` | 26 | responsive.css, style.css, home.css |
| `(max-width:767px)` | 11 | Sin espacio — mismo breakpoint |
| `(max-width: 767.98px)` | 4 | Bootstrap 4 estándar |
| `(max-width:767.98px)` | 4 | Sin espacio |
| `(max-width: 575px)` | 15 | |
| `(max-width:575px)` | 8 | Sin espacio |
| `(max-width: 575.98px)` | 7 | Bootstrap 4 estándar |
| `(max-width: 991.98px)` | 9 | |

**Problema**: El mismo breakpoint semántico se escribe con y sin espacio, y con valores `.98` vs sin decimales. Esto genera reglas que aplican en momentos ligeramente distintos (767 vs 767.98 = 0.98px de diferencia).

**Recomendación**: Estandarizar a la nomenclatura Bootstrap 4 (`.98px`) o a números enteros. Preferir `max-width: 767.98px` para consistencia con Bootstrap.

## 8. Conflictos Responsive

### 8.1 Breakpoints contradictorios

| Conflicto | Archivos | Efecto |
|-----------|----------|--------|
| `(max-width:767px)` vs `(max-width: 767px)` vs `(max-width: 767.98px)` | responsive.css, style.css, home.css, events.css, common-style.css | Mismo breakpoint semántico, 3 sintaxis distintas. Reglas que deberían aplicar juntas aplican en momentos ligeramente distintos. |
| `(max-width: 991px)` vs `(max-width: 991.98px)` vs `only screen and (max-width: 991px)` | múltiples archivos | Inconsistencia de 0.98px y presencia/ausencia de `only screen` |
| `@media (max-width: 575px)` vs `@media (max-width:575px)` | style.css, responsive.css | Solo diferencia de espacio — sin impacto funcional pero inconsistente |
| `(min-width:576px)` vs `only screen and (min-width: 576px)` | Bootstrap vs custom | Bootstrap usa sin `only screen`, custom a veces lo agrega |

### 8.2 Reglas móviles potencialmente pisadas

En el frontend, el orden de carga es:
1. `style.css` → define estilos base + mobile-first
2. `responsive.css` → pisa con media queries

Si una regla en `style.css` tiene un `@media (max-width: 767px)` y `responsive.css` tiene `@media (max-width: 767.98px)` para el mismo selector pero con diferentes valores, la de `responsive.css` gana por cargar después **y** tener un breakpoint 0.98px más amplio.

## 9. Tokens y Diseño

### 9.1 🔴 CRÍTICO: 6 verdes diferentes para el mismo propósito semántico

| Color | Usos | Ubicación | Propósito |
|-------|------|-----------|-----------|
| `#25d06f` | 173 | admin (atlantis.css + main.css) | Success primario admin |
| `#31CE36` | 78 | admin-skin.css (`--adm-success`) | Success skin override |
| `#10b981` | ~5 | admin-main.css (cards) | Card success |
| `#61dc96` | 8 | admin + frontend | Success alternativo |
| `#23dc73` | 7 | admin | Otro verde |
| `#15803d` | 8 | frontend (event-detail, style) | Verde oscuro frontend |

**Problema**: Seis valores hexadecimales distintos para "verde de éxito". En un design system, debería haber UN solo token `--color-success`.

**Impacto**: 
- Inconsistencia visual entre botones de éxito en admin (¿son `#31CE36` o `#25d06f`?)
- Si se cambia el tema a "tangerine", hay que rastrear y reemplazar 6 colores distintos

### 9.2 Colores fuera de sistema (top 10 hex en CSS custom)

| Color | Usos | ¿En Design System? |
|-------|------|-------------------|
| `#25d06f` | 173 | ❌ No documentado |
| `#ffffff` | 183 | ✅ Blanco base |
| `#1572E8` | 90 | ❌ Azul no documentado |
| `#31CE36` | 78 | ✅ `--adm-success` |
| `#F25961` | 76 | ❌ Rojo no documentado |
| `#6861CE` | 70 | ❌ Púrpura no documentado |

**Análisis**: Atlantis theme define su propia paleta (`.bg-primary`, `.bg-secondary`, `.bg-info`, etc.) con colores que no coinciden con el Design System Modern SaaS de TukiPass. Los colores `#1572E8`, `#6861CE`, `#FFAD46`, `#48ABF7`, `#F25961` son de Atlantis y no están documentados en el design system de TukiPass.

### 9.3 Border-radius inconsistentes

**Admin**:
- `admin-main.css`: `8px`, `14px`, `16px`, `50%`, `50px`
- `admin-skin.css`: `7px !important`, `8px !important`, `6px !important`, `12px !important`, `var(--adm-radius)` = `8px`, `var(--adm-radius-lg)` = `12px`

**Problema**: `admin-main.css` usa `14px` y `16px` para bordes, pero el design system define `8px` y `12px`. El `border-radius: 7px !important` en admin-skin.css no coincide con ningún token.

### 9.4 Prefijos vendor legacy

`atlantis.css` contiene 92 ocurrencias de prefijos `-ms-`, `-o-`, `-moz-` que ya no son necesarios en navegadores modernos (2026):
- `-ms-` (IE/Edge legacy) — totalmente obsoleto
- `-o-` (Opera Presto) — obsoleto desde 2013
- `-moz-` (Firefox antiguo) — muchos ya son unprefixed en Firefox moderno
